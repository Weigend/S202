LevelCalculator – Evolution History

From the first level view to the deterministic algorithm

v 0.1 2026-01-18 SCCDAG Foundation — Tarjan, LayerAssigner, SCCDAGBuilder
Key Fact: First complete SCC infrastructure in Java — TarjanSCCFinder, StronglyConnectedComponent, SCCDAGBuilder and LayerAssigner are established as core building blocks. Cycles are detected, the SCC-DAG is constructed, and levels are assigned to DAG nodes via longest-path.

Advantages

  • Solid algorithmic foundation (Tarjan O(V+E))
  • Cycles are explicitly modelled as SCCs
  • DAG longest-path yields correct layering depth

Limitations / Open

  • Analysis infrastructure only — no UI integration yet
  • Cycles are not broken — all SCC nodes land on the same level
  • No concept for packages vs. classes

v 0.3 2026-01-23 – 26 SCC Visualization + Heuristic SCC Breaker (In/Out-Degree)
Key Fact: Cycles are broken for the first time: HeuristicSCCBreakingStrategy selects back-edges based on in/out-degree scores. SCCs are highlighted in orange. A new strategy factory makes the breaking method pluggable.

Advantages

  • Cycles now yield a meaningful hierarchy instead of a flat world
  • Strategy pattern: breaking method is interchangeable
  • SCC visualization with back-edge highlighting
  • Zoom support (2–300 %)

Limitations / Open

  • Heuristic: equal scores → non-deterministic results
  • Purely class-based — package topology ignored
  • No formal rule set for violations

v 0.4 2026-02-18 Local UI Levels — DistrictRowLevelCalculator (2-phase pipeline)
Key Fact: Introduction of local levels per package: siblings (classes + sub-packages) within a package receive their own local levels from their sibling graph. Global levels remain for analysis, local levels drive the layout position. First 2-phase pipeline.

v 1.1 2026-05-07 Weighted Package Graph + Architecture-Anchored SCC Breaker
Key Fact: Package levels are now computed from a weighted inter-package graph: edge weight = number of classes in P_A's subtree that depend on at least one class in P_B (method-call counts). Ancestor propagation. SCC breaking uses weight-based rank scores instead of a pure degree heuristic. Simultaneously: ArchitectureAnchoredSCCBreakingStrategy uses external SCC-DAG anchors instead of internal degree scores.

Advantages

  • Package levels semantically grounded in real usage intensity
  • Ancestor propagation: parent packages inherit sub-package dependencies
  • SCC breaking guided by architectural context rather than topological chance
  • R3 rewritten: checks weighted package edge instead of container level

Limitations / Open

  • Two competing strategies (heuristic vs. arch-anchored) in production
  • Threshold (0.1) for symmetry detection still present → non-deterministic
  • DistrictRowLevelCalculator removed — was silently contradicting DomainModel

v 1.3 2026-05-15 Split: architectureLevel vs. localLayerIndex — ADR, 7-step migration
Key Fact: The single level field previously meant two completely different things. An ADR formalises the split: architectureLevel = global dependency-chain depth (analysis, violations), localLayerIndex = local sort key within the parent package (layout). 7-step migration completed in one day.

Advantages

  • Conceptual clarity: analysis and layout are separate concerns
  • childPkgUsedLevel lift and ContainmentEdge workaround eliminated
  • ArchitectureView rendering bug (view rendered below its renderers) fixed
  • Violations check architectureLevel; renderer uses localLayerIndex

Limitations / Open

  • Two level concepts require mental discipline to keep separate
  • localLayerIndex not yet wired into the invariant checker (added in step 7)

v 1.5 2026-05-24 SCC Breaking Along the Package Architecture — hypothesis drives class ordering
Key Fact: Pipeline order reversed: package levels are now computed first, then used to guide class SCC breaking. An edge A→B inside a class SCC is cut when pkgLevel(A.pkg) < pkgLevel(B.pkg) — it runs against the package flow. Same-package SCCs fall back to the degree heuristic.

Advantages

  • Class hierarchy follows the package architecture — conceptually correct
  • Back-edges are architecturally motivated, not topologically accidental
  • classBackEdgeKeys stored in DomainModel for R1 consistency

Limitations / Open

  • Degree fallback for same-package SCCs still heuristic
  • Threshold (0.1) in package SCC breaking not yet removed

v 2.0 2026-05-29 – 31 Deterministic Algorithm — no heuristics, no threshold, −6 % wrong-direction edges
Key Fact: The threshold (0.1) for symmetry detection and the entire SCCBreaker rank-score approach are replaced by a fully deterministic, threshold-free procedure: all edges running against the dependency flow are cut in a single pass. Symmetry is detected topologically (equal ranks). Case B (equal package level): all internal SCC edges are removed; levels emerge from external dependencies. Cross-platform verified on Minecraft 1.19.2 (Java ↔ .NET: 100 % identical levels).

Result on Minecraft 1.12.2: 3842 instead of ~4100 wrong-direction edges (−6 %)

Advantages

  • Fully deterministic — same input yields identical levels on Java and .NET
  • No magic threshold constant
  • ~380 lines of heuristic code removed (LocalLevelCalculator)
  • Performance: 60 s → 3–4 s on Minecraft input
  • SCCBreaker @Deprecated — Strategy pattern no longer needed
  • −6 % wrong-direction edges on Minecraft compared to v1.x

Limitations / Open
